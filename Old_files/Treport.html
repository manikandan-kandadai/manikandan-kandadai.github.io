Term Report

UX/UI Design Case Study

|




Overview

Problem:  Helping researchers and scholars to record and share their terminal session to the IT services.

Project Type:

Team (UM LSA IT services)

Role:

UX/UI Engineer, User Privacy Advisor.

Tools Used:

Sketch, HTML, CSS, JS.

Duration:

Jan 2017 - Present

Note:

This Project is a work in progress. We are constantly looking out for user feedback and iterating over the design.

Discovery

Target Users: Researchers and Scholars using Super computers at UM LSA and the LSA IT Department

Understanding the workflow:

Step1: The target users, logs into a web portal that provides access to an allocation of specific cores of Super Computer for them to use for the research.
Step 2: The Users then logs into a terminal session to import required modules in order to run experiements on the super computer.
Step 3: If they have trouble accessing certain functionality they call the IT support for help.
Step 4: The IT support will try and replicate the issue/bug and respond to the user either via email/call.



User Pain Points

Scholars and Researchers:
1) The Researchers and Scholars are not aware of things happening under the terminal session at a system level
2) They are not able to articulate the problem to the LSA IT in technical terms.
3) Unable to provide system information to LSA IT when asked.

LSA IT Services:
1) Unable to replicate the bug due partial information
2) Have a tough time getting system level information from the Researcher's terminal session.
3) Need a step-by-step tutorial to help the use or has to be physically present to help the Researcher.

Design

Design Philosophy: An intutive solution that provides the required information to the LSA IT services while keeping the user involvement minimal.

Tools: Sketch, Pencil Sketching, HTML, CSS, JS.

User Needs Evaluation:

The LSA IT was aware of the user problem they were facing since they have been getting a lot of emails about this issue. We started evaluating the needs by talking to Researchers and Scholars. We conducted interviews with them to understand their difficulties and I also spoke to LSA IT staff members to understand their side of the story.

It was apparent at this stage that they needed a common platform for users to report issues from the terminal session and the LSA IT to respond without asking further technical details from the users.

Personas

Two personas were sketched out after the interviews and needs assessment. These personas were then used to sketch few design ideas and validating the use with LSA IT.

The Proof of Concept

A proof of concept was developed along with few backend developers from LSA IT. A python client was written to record the terminal session from the user's machine and send it to a central web server where a webapplication was listening. A quick and dirty version of the web application frontend was coded to test the user experience and working.

Changes that Matter:

Once the feasibility of the idea was explored, it was time to add functionalities that targets the different needs of the users. I started asking questions to the LSA IT services staff to better understand the type of information that will be needed for them to understand a bug reported. The following are some of the features they wanted to see:
1) Know who the user is and a brief description of the bug.
2) Get access to system environment variobles and log files.
3) Be able to send a recorded terminal session as a reply to the user's request.
4) Did not want a new ticketing system since they already had one in place.
5) would prefer to document all service requests through the existing ticketing system.

Ideas in Action:

After understanding the features that the LSA IT felt usable for them, I started sketching out the UX of the various features and their placement on the screen. I validated the UX by showing it to few IT staff and observing their reaction and emotions towards the UI.

Design Decisions:

1) Presenting System level information:
It was important to present information about the user's terminal session and the system properties to the IT services in a way that is very obvious.
This information will help the IT services to help them prevent asking Users technical questions.
2) Log files:
A downloadable link to log files is also provided if the IT service wants to have a closer look at the log files generated by their system tasks to better understand the bugs.
3) Generating Sharable URL:
This feature helps the IT service to upload a solution in a playable video format that shows the users how to solve the bug. This video is shared on a seperate sharable window and the link to that will be emailed by the IT services from the existing ticketing system.

Next Steps:

The project is a work in progress and we envious the future to be a lot of user testing by piloting the product in different phases and iterating over the user test results. Once we have generated enough feedback we will rework on the usability and UX of the web application.

Conclusion


This was a really interesting project for me since it tackled an issue that all IT services face. This project pronunced the fact that different users of the same application will have different skills and abilities when it comes to technical knowledge and hence it is really important to bridge this gap by making the solution accessible to both ends of the spectrum.
