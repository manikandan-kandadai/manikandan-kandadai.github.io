    Parsing JSON Logs

    UX & UI Design of feature in RSA NetWitness platform to parse JSON log sources

    Overview

    The Problem

    System administrators continually onboard new log sources to the RSA NetWitness SIEM application and need a more efficient interface to parse the structured logs and map it to searchable meta keys.

    The Team

    This project was done by a team of 2 designers including me. In addition to designing, my contribution also included performing UX Research - Usability Testing and Interviews


    Full case study is available upon request.

    About the problem

    # What is log parsing?

    RSA NetWitness is a SIEM (Security Information and Event Management) that aggregates and analyses security events across the entire IT infrastructure. SIEM collects security data from network devices, servers, domain controllers, etc. from logs and packet data to identify threats. To make the logs searchable in the platform, it needs to be recognized and mapped to meta keys (searchable keys optimized for performance). This process of recognizing and mapping log data to searchable meta keys is called parsing.

    # Who are the users?

    Primary users of this system are system administrators responsible for integrations and maintaining the system's health and wellness.

    # The Objective

    The objective is to design an interface that would enable users to easily parse structured JSON logs and extend the current UI's capabilities to support more advanced use cases.

    UX Process




    # Whats Changed?

    We followed the double-diamond process of UX. However, we did change the process to accommodate our needs. The following is the process that worked for us.

    1. Perform upfront research to understand requirements and workflow (SME or Users)
    2. Utilize design reviews for feedback and critiques (Designers & PM)
    3. Final Wireframe or Prototypes - Usability Testing with Users

    Log Parser Workflow (Backend)




    # User Pain points

    The legacy way of doing it was to edit the log parser config files in a log decoder to change how the logs are parsed for different log types. This approach has several disadvantages

    1. Bulk changes can be unmanageable
    2. The config files are prone to human error, and this can lead to unexpected results
    3. New parsers would require more thought while onboarding.


    # Challenges & Constraints

    There were several technical constraints and trade-offs that were made while designing the interface. The following are some of the significant constraints for this project.

    1. Multiple log types - logs can be of any kind, each requiring a change in the backend data structure. This resulted in the need for the UI to be slightly different for handling different types.
    2. To identify an MVP that works with the timeline. This involved identifying the minimal usable features that can be released.

    # Single and Multi-select Interactions


    In the mapping mode, clicking on a specific mapping on the right will highlight its corresponding Key-Value pair on the left. This is true for the other way around as well.
    The highlight color is the default selection color indicating that the highlighted Key-Value pair is being actively worked on. Once a key has been configured, the color changes to green to indicate that it is modified.

    Highlighting behavior

    If users want to select a single Key/Value in the log message and tag them, they can click on the Key or Value and click on 'Add Selection to Mappings' button.

    Single Select behavior

    This is true for multi-select as well. Holding down shift or ctrl/cmd and clicking on the rows will enable a multi-select option. This feature is handy if users want to tag only specific keys and don't want to auto-extract all the keys.
