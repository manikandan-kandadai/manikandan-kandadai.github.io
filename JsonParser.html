    Parsing JSON Logs

    UX & UI Design of feature in RSA NetWitness platform to parse JSON log sources

    Overview

    The Problem

    System administrators continually onboard new log sources to the RSA NetWitness SIEM application and need a more efficient interface to parse the structured logs and map it to searchable meta keys.

    My Contribution

    I made incremental UX and visual updates to an earlier version of the feature to add more functionalities.


    Full case study and final visual designs of version 2 is available upon request.

    About the problem

    # What is log parsing?

    RSA NetWitness is a SIEM (Security Information and Event Management) that aggregates and analyses activities across the entire IT infrastructure. SIEM collects security data from network devices, servers, domain controllers, etc. in the form of logs and packet data. This data is used to analyze and identify threats in the system. One key component in this is the ability to ingest logs from different sources. In order to make the logs searchable in the platform, it needs to be recognized and mapped to meta keys (searchable keys optimized for performance). This process of recognizing and mapping log data to searchable meta keys is called parsing.

    # About the feature

    Logs come in all different shapes and sizes. Some logs are plain text; some are structured (CSV, JSON, XML, etc.). The requirement is to design a feature that would enable system administrators to parse structured JSON logs. The legacy way of doing this was to manually edit the configuration files of the log decoders (a gateway for ingesting logs). This was very inconvenient and prone to human errors resulting in unexpected outcomes. The UI makes this process more convenient for users to configure new and existing parsers.

    # Who is the user?

    Primary users of this system are system administrators responsible for integrations and maintaining the system's health and wellness.

    # The Objective

    The objective is to design the version 2 interface that would extend the current UI's capabilities to support more advanced use cases.

    UX Process


    # Not a typical Double Diamond



    In the above double diamond process, my work fits the rear end of the process as the problem definition and brainstorming solutions were done as a part of the version-1 implementation. For version-2 (which I was a part of), I made incremental changes to already established patterns and interactions. The designs and interactions were already tested and iterated over. Since the incremental updates needed minor changes, no new user research was performed instead, feedback from SMEs were taken into consideration. This vastly increased our speed and efficiency.

    Challenges & Constraints


    Working with the version-1 UI design proved to be helpful as it was already validated for usability. However, this also meant that there was a fundamental constrain in the layout around which new features need to be incorporated. This constraint allowed me to develop different design options to incorporate advanced features without making layout changes to already developed version 1.
