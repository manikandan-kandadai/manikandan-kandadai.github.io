    Cyber Threat Hunting UI

    Web UI Design for cyber security analysts to hunt for security threats.

    Overview

    The Problem

    Cyber Threat hunters are continually sifting through various data to uncover potential threats supported by their hypothesis. They need an Interface that would enable them to explore data and validate this hypothesis.

    Disclaimer

    This is a concept UI inspired by the complexity of workflow in performing threat hunting


    About Cyber Threat hunting

    Cyber Threat hunting involves proactively looking for security threats that evade traditional defense mechanisms. This requires specialists to explore and sift through various data to validate certain hypotheses on potential threats in their environment.

    Translating Hypothesis


    # Querying

    Once a hypothesis has been formed, the next step is to transform that into queries that the system can understand. Most products query the underlying data directly (searching for a specific value). This approach may be quick and efficient for the database, however this creates two user problems.

    1. New analysts have to learn and remember underlying data to query the system and validate their hypothesis
    2. Querying the data directly means users will have to translate their hypothesis into queries (that the system can understand) on their own. This is an additional mental effort which can be avoided and better used for analysis

    # Solution

    One possible solution is to have an analysis layer that is abstracted away from the underlying data. This higher-level layer will translate the hypothesis (in a human language) into queries that the backend would understand. Having an analysis layer would enable users to query malicious behaviors directly rather than looking for specific data points that would suggest the presence of a threat.


    This concept is not new, the above query language is similar to cypher query language which uses a graph database.

    Hunting


    # Multiple leads


    The Problem

    Once a hypothesis is defined and queried for, there are multiple ways an analyst can look at the data. This poses an exciting workflow challenge for designing. The primary objective here is to design an interface that would allow users to explore the data starting from any lead.

    Solution

    The image on the right lists all potential leads grouped under their entities (Users, Files, Network, Process). This is always available on the interface and can be used by analysts to mix and match them while hunting. This list is also dynamic, reacting to the queries that are running. A new lead can be added to a query at any time by clicking on the icon.

    When combined with an interface that allows for adding various leads, the analysis layer is a powerful toolkit that allows for quick and more efficient triage.



    # Dealing with Noise


    The Problem

    Once the query returns a result, the data can be inundated with noise. There could be data about a real threat (signals) buried under thousands of other data points. This is like finding a needle in a haystack. It is essential to enable users to identify and differentiate those signals from noise.

    Solution

    A lot of Machine Learning models are under development to model user behavior across data. This behavior modeling would help identify real threats from false positives. One approach to deal with this is entropy-based anomaly detection. Entropy is the measure of randomness in data. If there is an encoded payload in network traffic, the entropy value will be different from normal network traffic. This subtle change in entropy can be identified with precision using machine learning techniques. When implemented well, entropy can be a great toolkit to reduce the noise and elevate signals.

    This entropy chart highlights a correlation between abnormal network and file process activity that spike simultaneously. When Machine learning models are employed, it can detect these spikes and highlight the correlation between network and file activities.

    Indentify Threats


    # Working with the data


    Machine learning models jump-start the analysis part but elevating signals from noise. The signals are now available to be closely examined by analysts by digging through data and eliminating false positives. At this stage, users need an efficient way to filter and look at hundreds and thousands of data points to determine if the activity signifies a real threat.

    The problem

    The complexity of analysis lies in that every data connects to every other data point through a non-obvious relationship. These relationships are incredibly vital for threat hunting as it sheds more context on the signals. The problem is that the data could be spread across different time periods, making the analysis even harder.

    Solution

    Fortunately, there are frameworks such as the MITRE Att&ck matrix that helps identify anomalous behaviors. When combined with other contextual information (Enrichments), this framework can help qualify the observed behavior into normal or anomalous behavior. The enrichments can be from internal or external sources.

    The above visualization helps analysts further explore the relationships between the queried parameters. The enrichments and other qualifying information are overlayed on top of the data points to provide more context. Users can turn this off at any point in time if needed. The visualization is also dynamic and interactive. Users can choose to look at all data points or select portions of visualizations for more information.

    # Why Parallel Co-ordinates

    The choice of parallel coordinates as a visualization technique is because of its advantages for multi-variate data analysis over other charting types.

    1. With parallel coordinates, each axis can have a different scale and point to relationships across different variables. Parallel coordinates also scale well for multiple variables.
    2. The only downside of using parallel coordinates is that the visualization can get to an unusable stage with plenty of data points. However, this is no issue in this situation as the data is already reduced to a limited set through entropy-based Machine Learning techniques.

    Final Solution






    # How it all works together

    The entire UI design works as one and adds value to the threat hunting workflow. Once a hypothesis is queried for, users can start and explore data. If some users do not want to work with graphs and charts, they can always disable and collapse them to work directly with the data table if that suits their workflow.

    # Caveats

    1. This is a concept design and hence not validated for any technical accuracy and feasibility
    2. The mockups were designed based on research and feedback from a few security researchers. However, more research is needed to validate assumptions and constraints
    3. Performance of this interface can be questionable given the massive amount of data d3 will work with.
